# Home Assistant Integration

Primary integration is ESPHome native API. MQTT is supplementary and optional.

## Entity Naming

Use a stable device prefix such as `greenhouse_controller_*`.

Examples:

- `greenhouse_controller_high_air_temperature`
- `greenhouse_controller_window_position`
- `greenhouse_controller_manual_mode`
- `greenhouse_controller_network_mode`

## Expected Entity Groups

### Sensors

- high air temperature
- high air humidity
- low air temperature
- low air humidity
- intake air temperature
- soil moisture
- flow rate
- estimated window position
- uptime
- optional Wi-Fi RSSI when relevant
- irrigation cooldown remaining
- irrigation settle delay remaining
- irrigation run time

### Binary Sensors

- door state
- Ethernet connected
- Wi-Fi connected
- AP mode active

- irrigation fault active

### Buttons / Actions

### Numbers / Selects

- ventilation open threshold
- ventilation close threshold
- soil moisture minimum threshold

## Integration Notes

- Home Assistant availability must not disable local automation
- unavailable sensors should show unavailable, not stale-valid values
- diagnostics should make recovery states explicit

## Current Implementation Notes

- native API is the primary Home Assistant path
- MQTT is configured as supplementary telemetry and should not be treated as the source of truth for control
- the local ESPHome web server is currently the first-pass local maintenance surface
- display entities are configuration placeholders until the exact display hardware is chosen
- entity names currently follow the device-friendly-name pattern generated by ESPHome; object IDs should remain stable through the device slug
